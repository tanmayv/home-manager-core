# Agent Tracker Cross-Platform Plan

## Goal

Make `agent-tracker` work on Linux and macOS by moving the tracker from a procfs-driven model to an identity-driven model.

## Why this change

Current Linux-only assumptions:

- `systemd.user.services.agent-tracker`
- `/proc`-based process identification in `rpc_handler.py`
- `/proc`-based liveness checks in `monitor.py`
- `/proc` reads in hook scripts for caller identification
- hard-coded `/tmp/agent-inboxes`

We want the tracker to keep the current tmux UX and inbox behavior, but make explicit identity the source of truth.

## Core design

### Identity model

Primary key:
- `agent_id` — stable UUID generated by `agent-wrapper` at agent startup

Secondary / mutable fields:
- `agent_name` — display name, may change via rename
- `tmux_pane`
- `tmux_socket`
- `session`
- `agent_type`
- `agent_cmd`
- `status`
- `waiting_approval`
- `last_heartbeat`
- `pending_notifications`

### Invariants

1. `agent_id` is the authoritative identity in tracker state and RPC.
2. `agent_name` is user-facing and mutable; rename never changes `agent_id`.
3. tmux metadata stores both `@agent_id` and `@agent_name`.
4. RPC calls coming from agents should carry explicit `agent_id`; peer credentials and procfs are optional hardening/fallback only.
5. Wrapper heartbeat is the primary liveness signal.
6. Wrapper-exported `AGENT_NAME` is not authoritative after initial registration.
7. Pane disappearance causes immediate cleanup.
8. Pane existence alone is not proof that the agent is still alive.
9. All runtime paths must be derived from `${XDG_CACHE_HOME:-$HOME/.cache}/agent-tracker`; no hard-coded user home paths.

## Environment contract

`agent-wrapper` will export at least:

- `AGENT_ID`
- `AGENT_NAME`
- `AGENT_TRACKER_SOCKET`
- `TMUX_PANE`
- `TMUX_SOCKET`
- `AGENT_TYPE`
- `AGENT_CMD`

Hooks and helper scripts should use these values first. tmux lookup should be fallback only.

Important mutability rule:
- `AGENT_NAME` in the wrapper environment is not authoritative after initial registration
- rename changes tracker state and tmux `@agent_name`, but does not rewrite the wrapper environment
- heartbeat must identify only by `agent_id` and liveness fields
- if re-register needs the current display name after tracker restart, it should read `@agent_name` from tmux metadata or use the tracker-assigned name, not a stale wrapper env var

## tmux metadata contract

Store on each agent pane:

- `@agent_id`
- `@agent_name`
- `@agent_type`
- `@agent_cmd`

Authoritative meaning:

- `@agent_id` is authoritative for identity recovery
- `@agent_name` is display-only and may be changed independently

## RPC direction

Target steady-state RPC methods:

- `register`
- `heartbeat`
- `update_agent`
- `rename`
- `send_message`
- `get_inbox`
- `whoami`
- `unregister`
- `list`

Expected identification precedence:

For agent-originated RPC:
1. explicit `agent_id` in params
2. `AGENT_ID` from environment
3. explicit `tmux_pane` if uniquely mapped in tracker state
4. `@agent_id` from tmux metadata on the current pane
5. optional Linux-only peer-cred/procfs fallback during migration

For human/CLI target lookup:
1. explicit target `agent_id`
2. explicit target `agent_name` if uniquely mapped
3. explicit `tmux_pane` if uniquely mapped

`agent_name` should never win over an available explicit or environment-provided `agent_id`.

## Startup / lazy-start contract

Lazy-start is part of the core design, not follow-up.

### `ensure_tracker_running()` behavior

Used by `agent-wrapper` and `agent-tracker-ctl`:

1. try connecting to the socket
2. if socket works, continue
3. acquire the startup lock
4. re-check socket after taking the lock
5. if the socket file exists but still does not accept connections while the lock is held, treat it as stale and remove it
6. if socket is still absent after stale cleanup, start the daemon
7. wait briefly for socket readiness with retries
8. if daemon still does not come up, fail gracefully

### Lock primitive

The startup lock should be implemented in Python using `fcntl.flock` on a lockfile under the tracker cache dir.

Reason:
- available on Linux and macOS
- avoids dependence on shell `flock`
- easy to share between wrapper helpers and CLI helpers

### Race handling

- only the process holding the lockfile starts the daemon
- others wait for socket readiness
- simultaneous wrapper startup should not create duplicate daemons

### Failure behavior

If tracker is unavailable:
- wrapped agent should still run
- tmux pane title/name may still be set locally if possible
- tracker-specific status/inbox features degrade gracefully

## Heartbeat / liveness policy

### Heartbeat producer

Primary producer:
- `agent-wrapper`

Reason:
- wrapper survives even when hook events are sparse
- avoids relying on tool-specific event frequency

### Policy

- heartbeat interval: 5s
- stale threshold: 20s
- cleanup grace period: 30s
- stale-confirmation window: 10s (two consecutive monitor passes by default)

### Heartbeat implementation

The heartbeat loop lives in a small background child process started by `agent-wrapper` after successful registration.

Behavior:
- sends `heartbeat(agent_id, tmux_pane, tmux_socket, session)` every 5s
- does not treat wrapper-exported `AGENT_NAME` as authoritative after initial registration
- is terminated by the wrapper's normal cleanup path
- if the daemon restarts or replies `not found`, the helper runs `ensure_tracker_running()` and re-registers using the same `agent_id`
- re-register reads the current display name from tmux `@agent_name` metadata if needed, rather than from a stale wrapper env var
- heartbeat re-registration is an upsert, not a new logical agent

### Cleanup rules

An agent is removed immediately when:
1. its tmux pane no longer exists

An agent is marked `stale` when:
2. heartbeat is older than 20s while the pane still exists

A `stale` agent is removed after the 30s grace period if, during the stale-confirmation window, pane inspection shows no live non-shell agent process attached to the pane tty.

Cross-platform pane inspection rule:
- inspect the pane tty using `ps -t`
- if only the shell or known transient helper commands remain, treat the pane as no longer hosting the agent
- if a plausible long-lived agent process remains, keep the record and continue waiting for heartbeat recovery
- the shell/transient allowlist should be shared with the existing discovery heuristic so liveness cleanup and process discovery classify commands consistently; this list must be validated on both Linux and macOS to avoid false removals

This rule intentionally handles abnormal exits such as wrapper crash or `kill -9` with the pane left open: stale tmux metadata is eventually evicted even if the pane still exists.

Hook events are supplemental:
- they update status like `working`, `waiting`, `idle`
- they are not the primary liveness signal

## Recovery after tracker restart

Recovery will no longer promise exact PID reconstruction.

### Recovery source

- tmux pane metadata (`@agent_id`, `@agent_name`, `@agent_type`, `@agent_cmd`)

### Recovery behavior

If tracker restarts and pane still exists:
- recover `agent_id`
- recover `agent_name`
- recover pane/session/type/cmd
- initialize status as `unknown`
- initialize `waiting_approval = false`
- preserve inbox data from on-disk storage
- treat `pending_notifications` as volatile for MVP; they are not persisted across tracker restart unless explicit persistence is added in a later phase

First heartbeat transitions the agent back to live state. Pending notifications are not flushed while status is `unknown`; they flush only after the first explicit transition to `idle`.

Focus acknowledgment semantics:
- the existing pane-focus acknowledgment rule applies only to agents in explicit `waiting` state
- recovered `unknown` agents do not transition to `idle` merely because their pane is focused

If pane is gone:
- do not recover the agent

## Storage paths

Avoid hard-coded `/tmp/agent-inboxes`.

Proposed base path:
- `${XDG_CACHE_HOME:-$HOME/.cache}/agent-tracker`

Under that:
- socket: `agent-tracker.sock`
- inbox dir: `inboxes/`
- lockfile: `agent-tracker.lock`
- optional logs/state snapshots under same base

This works on Linux and macOS and keeps tracker state colocated. Implementations must derive these paths dynamically from the environment-backed base path rather than hard-coding a user-specific home directory.

## Existing behavior parity

We want to preserve these current behaviors:

1. self rename
2. force rename of another agent
3. inbox send/read
4. busy-agent pending notification queue
5. pending notification flush when agent returns idle
6. user acknowledgment by focusing a pane: `waiting -> idle`
7. pane title sync with agent name
8. tmux option sync for agent metadata
9. unregister on pane close
10. tracker restart recovery
11. spawn timeout cleanup
12. `whoami` / `list` UX

### Intentional behavior changes

1. after tracker restart, recovered agents may show `status = unknown` until heartbeat resumes
2. PID-based exact recovery becomes best-effort/diagnostic, not core correctness
3. procfs becomes optional Linux fallback rather than required behavior
4. pending notifications are intentionally volatile across tracker restart in the MVP unless later persistence is added

## Register / conflict semantics

### Register idempotency

- `register(agent_id=existing_id, ...)` is an upsert
- if the same `agent_id` registers again, the tracker refreshes session/pane/socket/name/type/cmd/wrapper metadata and `last_heartbeat`
- re-registering the same `agent_id` must not create a second logical agent

### Pane conflicts

- if a pane already maps to a different `agent_id`, the new registration wins for that pane
- the old record is marked `stale`, its tmux metadata is cleared best-effort if it still owns another pane association, and it is removed by normal cleanup

### Agent move / reconnect

- if an existing `agent_id` re-registers from a new pane, the tracker treats it as the same logical agent moving/rebinding
- the new pane becomes authoritative
- old pane metadata is cleared best-effort

## Name-index semantics

- `agent_name` must be unique among live and recovered tracker entries
- recovery reserves names before the daemon begins serving user RPC
- if a requested registration name is already taken by another `agent_id`, the new registration receives a suffixed name
- rename to an already-occupied name fails unless the target name is already owned by the same `agent_id`

## Open implementation notes

- keep a name index so CLI commands can still target human-readable agent names
- rename must update tmux title and `@agent_name`, but never `@agent_id`
- `send_message` should store inbox by `agent_id`
- `whoami` should return both `agent_id` and `agent_name`
- during migration, dual support for `agent_name` and `agent_id` RPC params may be useful
- self-identification on macOS should prefer explicit `agent_id` params/env over tmux name matching
